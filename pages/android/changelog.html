Changelog

September 01, 2016
  • New methods are added to the parameter options to fetchIndoorPOIsForBuilding and fetchOutdoorPOIsForBuilding, in order to filter the received pois by the custom fields.
August 30, 2016
  • READ_EXTERNAL_STORAGE and WRITE_EXTERNAL_STORAGE permissions are eliminated. Now are optionals.
August 29, 2016
  • Now Point2f and SitumFloorPoint classes implement the Parcelable interface.
August 25, 2016
  • Two new methods are added in SitumBaseLogin:
    • setDashboardUrl(): allows to change the Dashboard URL.
    • setDefaultDashboardUrl(): changes the Dashboard URL to the SDK’s default value.
  • Method getDashboardUrl() is added in SitumBaseDataManager to obtain the Dashboard URL.
  • The Dashboard URL’s access is eliminated from the Shared Preferences.
August 24, 2016
  • BUILDING-WITH-NO-CALIBRATIONS error is added in SitumError.
August 22, 2016
  • The possibility to adjust the position a route during navigation is added.
  • The following methods are added in SitumIPSManager class:
    • adjustLocationToRoute(list<SitumFloorPoint>): starts setting the next localizations received to a route.
    • stopAdjustingLocation(), for setting the localizations to the route.
    • isAdjustingLocation(): it returns true if the setting localizations to a route is activated
July 28, 2016
  • If proguard is used to obfuscate the apk’s, it is required include the following line into the proguard-rules:
  • keep class es.situm.sdk.v1.** { *;}
July 27, 2016
  • The method of the SitumIndoorPOI.getLevelid() class is renamed to SitumIndoorPOI.getLevelId().
  • The method of the SitumPOI.getHtmldescription() class is renamed to SitumPOI.getDescription().
July 22, 2016
  • A method is added in SitumBaseDataManager to invalidate the cache: invalidateCache().
July 18, 2016
  • Calibration management is improved:
    • The SitumCalibration class is added: serves as a calibration wrapper.
    • All the methods that used route calibrations as String have changed to SitumCalibration:
    • getSavedCalibrations(), getCurrentCalibration(), uploadSelectedCalibrations(), deleteCalibrations()
    • A callback is added, UploadCalibrationCallback, used in methods uploadSavedCalibrations() and uploadSelectedCalibrations(), notifying the result of the calibration uploads.
    • The deprecated methods uploadSavedCalibrations() and uploadSelectedCalibrations() doesn’t have callback as a parameter.
    • The NetworkMessage class is added: it contains information in an onUploadError().
July 15, 2016
  • SitumRouteCalculator interface is modificated:
    • Now is necessary to pass a List as a parameter to float distanceToGoal(List<SitumFloorPoint>) instead of an ArrayList.
    • Now List<SitumFloorPoint> computeRoute(SitumFloorPoint, SitumFloorPoint, boolean) returns a List instead of an ArrayList.
July 13, 2016
  • Deprecated login methods in SitumBaseLogin are eliminated.
July 08, 2016
  • Different libraries are added to different architectures: arm64-v8a and armeabi-v7a.
July 06, 2016 - 1.64c version
  • Getter of SitumPOI is modified to return a Map instead of a HashMap.
  • Fully functional POIs categories are added:
    • Every single SitumPOI has a getCategory().
    • All categories can be obtained with fetchPoiCategories().
    • Icons from a category can be obtained with fetchPoiCategoryNormal() and fetchPoiCategorySelected().
  • Disk cache added to the category icon.
June 28, 2016
  • API cache is added in DiskRepository, allowing to specify a maxAge. If the internet connexion fails, the cache data are returned even if they are old.
  • getCacheMaxAge() and setCacheMaxAge() methods are added to the SitumBaseDataManager class to obtain and specify maximum cache time.
June 22, 2016
  • A temporary fix for certificates in version 4.2.2 (API 17) is added.
June 17, 2016 - 1.64 version
  • Custom fields (key-value) are added in POIs.
  • Info field is added in buildings.
  • The first version of the categories are added.
  • The SitumSearchManager is added to manage the searches in the models. At the present time, only exits one way to filter POIs according to this custom fields.
  • SitumRouteCalculator interface is modified:
    • Now is necessary to pass a List as a parameter to float distanceToGoal(List<SitumFloorPoint>) instead of an ArrayList.
    • Now List<SitumFloorPoint> computeRoute(SitumFloorPoint, SitumFloorPoint, boolean) returns a List instead of an ArrayList.
June 08, 2016
  • Context as a first parameter of SitumLogin is added, but for now it is only necessary for debug options, as in Inphone.
  • Used shared preferences to load the positioning and dashboard urls
  • SitumIPSManager and SitumPositioningManager are modified:
    • changePositioningServer() is eliminated.
    • Now getPositioningaddress() only returns in Debug’s mode.
May 10, 2016
  • Bluetooth deactivation by stopping the positioning is added, but only if the Bluetooth was activated when the positioning starts.
May 04, 2016
  • The two login methods are modified. Now it is necessary to pass a Context as a first argument.
  • getBuilding() method is added to SitumLevel, SitumPOI and SitumEventI, to obtain their corresponding building ID.
  • Floor images cache is added in fetchMapForLevel(). It is no longer necessary to implement in the APP.
April 28, 2016
  • A new method is added to the SitumLogin class to logout.
April 27, 2016
  • An error as a result of validating the callback that send locations when they reach the server is corrected. If the callback is null, none position is returned, and it can be null if a call to the method ipsmanager.start(building) is done before calling the setPoseReceived().
March 30, 2016
  • A new method is added to the SitumPoseReceiver interface. This new method, onInvalidPoseReceived(), only receives not valid positions. For more information, please consult the javadoc.
March 11, 2016
  • Two new methods are added to SitumIPSManager class:
    • 1. start(SitumBuilding building, SitumPositioningOptions options)
    • 2. start(SitumBuilding building)
  • start(SitumBuilding building, boolean usewifi, boolean useble, boolean usemag) becomes deprecated, and its use is not recommended. If the use of this method is:
    • ipsmanager.start(building, true, true, true);
  • it can be replaced by:
    • ipsmanager.start(building);
  • The result is exactly the same.
  • A new class is created: SitumPositioningOptions, allowing positioning server options configuration. For example: WiFi, BLE, or both of them, used for positioning. This class also allows inform the service positions will be simulated from a csv file. In this way, development and testing will be much faster. This csv format has to be:
    • time:x:y:yaw:level:validTransition:radius:hasOrientation
  • It is required to indicate all the fields, even if the value is 0. If the format is incorrect, the simulator will not be able to provide positions.
March 01, 2016
  • Android-23 compatibility is added. Two steps must be completed to run the APP in this platform:
    • 1. In the android {} plugin, which is in the project build.gradle, write:
    • useLibrary "org.apache.http.legacy"
    • 2. Update loopi dependence to its latest version:
    • compile 'com.loopj.android:android-async-http:1.4.9'
  • Both steps are done to maintain the code compatibility annotated with Deprecated.
February 29, 2016
  • The isValidUrl() method is added in SitumBuilding to check if the building has been calibrated or not. For more information, please consult the javadoc.
December 16, 2015
  • cancel() method is added in SitumCalibrationManager to stop a calibration without saving it in the disk.
December 11, 2015
  • The SitumBuilding class's toEarthAngle() method javadoc is corrected.
  • The exactly time in milliseconds is added to the SitumLocation class when a location arrives at the SDK from the server.
October 09, 2015
  • It is necessary to include the following permission to the AndroidManifest.xml to check the network state:
    • <uses-permission android:name="android.permission.ACCESS_NETWORK_STATE"/>
  • Error control is added in the connectivity. Now the system will not provide position if you try to positioning in a building without being physically there, reporting an error. Error codes are contained in SitumError class. This is an example about how to see the errors:
    • positioningManager.setSensorErrorListener(new SitumSensorErrorListener() {

  @Override
  public void onError(SitumError situmError) {
    if (situmError.errorId == SitumError.USER_NOT_IN_BUILDING) {
        // acciones a realizar cuando la persona esta intentando posicionarse
        // en un edificio en el que no se encuentra fisicamente
    } else if (situmError.errorId == SitumError.INTERNET_NOT_AVAILABLE) {
        // acciones a realizar cuando el dispositivo pierde la conexion a la
        // red
    }
  });

  • The login method is modified. There are two methods now to achieve a SitumDataManager class’s valid instance:
    • 1. Call the SitumLogin.login(email, password, <Callback>) method. If correct data is provided, callback will provide a SitumDataManager instance.
    • 2. Call the SitumLogin.login(email, apikey) method, getting directly a SitumDataManager.

July 15, 2015
  • Buildings now have a rotation to be visualized on the map in a realistic way. The following steps must be followed to paint the buildings correctly on the map:
    • 1. Have the buildings corners.
    • 2. Change the building rotation to grades.
    // Example
Point2f[] point = building.getDrawingPoints();
    LatLng topLeft     = new LatLng(point[0].getY(), point[0].getX());
    LatLng topRight    = new LatLng(point[1].getY(), point[1].getX());
    LatLng bottomRight = new LatLng(point[2].getY(), point[2].getX());
    LatLng bottomLeft  = new LatLng(point[3].getY(), point[3].getX());

    LatLngBounds.Builder latLngBuilder = new LatLngBounds.Builder();
    latLngBuilder.include(topLeft).include(topRight).include(bottomRight).include(bottomLeft);
    LatLngBounds esquinas = latLngBuilder.build();
    float rotacion = building.getRotacion() * 180 / Math.PI;
    googleMap.addGroundOverlay(new GroundOverlayOptions()
        .image(bitmap).positionFromBounds(esquinas).bearing(rotacion);
